# Reviewer Quickstart

This file is a short path through the PQCFirm artifact. It is intended for reviewers who want to reproduce the main claim checks without reading every script first.

## 1. Software replication

From the package root, run:

```bat
run_replication.bat
```

On Linux or macOS, run:

```bash
bash run_replication.sh
```

Expected status: `PASS` in `artifact/results/run_replication_summary.json`.

The run creates a Python environment, installs dependencies, verifies benchmark JSON files, regenerates the LOC count, recomputes the taxonomy statistics, recomputes the four-pass LLM-assisted annotation-consistency statistics, reruns PQCFirm on the ESP32-S3 benchmark, reruns the Mbed TLS raw and filtered triage studies, runs mutation testing, evaluates the curated seeded corpus, recomputes differential-test summaries, and checks the high-level claim matrix.

## 3. Optional hardware recapture

The main replication run does not require hardware. To regenerate the hardware logs, connect an ESP32-S3 board and run:

The hardware target for this artifact is ESP32-S3 only. Raspberry Pi mentions, if found by text search, come from bundled third-party upstream text or mined issue titles and are not PQCFirm hardware experiments.

```bat
run_hardware.bat COM8
```

Replace `COM8` with the correct serial port. On Linux or macOS, use:

```bash
bash run_hardware.sh /dev/ttyUSB0
```

Expected hardware evidence:

The wrapper scripts recapture the ESP32-S3 failure-reproduction logs and verify the existing paired-trial files. To recapture paired trials as well, use `python run_hardware_capture.py --port <PORT> --benchmark-timeout 300` without `--skip-paired`.

| Hardware check | Expected result |
| --- | --- |
| 8 KB stack failure-reproduction firmware | `Guru Meditation Error` |
| 96 KB stack failure-reproduction firmware | `[FAILURE_REPRO] TEST PASSED` |
| Paired trials | 100 hardware rows with 0 completion/status mismatches |
| N=30 repeatability subset | 30 hardware rows with 0 completion/status mismatches |

Without hardware, reviewers can still inspect the included serial logs and `artifact/results/paired_trials.csv`.

## 4. Where to inspect claim evidence

For a compact claim-to-evidence map, see:

```text
CLAIM_TO_EVIDENCE.md
artifact/CLAIMS_SUPPORTED.md
artifact/results/claim_matrix.json
```

`CLAIM_TO_EVIDENCE.md` is a reviewer-facing guide. `artifact/CLAIMS_SUPPORTED.md` and `artifact/results/claim_matrix.json` are regenerated by the verification harness.


## Optional additional evaluation

The artifact includes optional support for external-codebase scans of liboqs and wolfSSL after downloading pinned snapshots into `artifact/external_codebases/snapshots/`. Stored external-codebase summaries are included under `artifact/results/external_codebases/`; rerun `artifact/external_codebases/run_external_codebase_eval.py` after downloading pinned snapshots to reproduce them. The artifact reports the validated ESP32-S3 crash/safe failure-reproduction experiment as the hardware evidence.


External-codebase audit-note: audit rows are selected by stable SHA-256 ordering over rule, normalized path, location, and message. The TP/FP/REVIEW labels are generated by the documented deterministic triage function in `artifact/external_codebases/run_external_codebase_eval.py`; they are not random labels, not an online LLM call, and not an independent human manual audit.


### Windows PowerShell note

From PowerShell, run the batch wrapper with an explicit current-directory prefix:

```powershell
.\run_replication.bat
```

If a previous failed run left an incomplete `venv` folder, delete `venv` and rerun the command. The batch wrapper also attempts to detect and recreate an incomplete virtual environment automatically.
